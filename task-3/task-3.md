# Практическое задание №3

> **О чём задание:** проектирование архитектуры и логики приложения "Зелёный агент". Нужно выделить доменные зоны (DDD), описать критический сценарий в формате BDD, нарисовать wireframes ключевых экранов и спроектировать API-контракты для MVP.

## DDD (Domain-Driven Design)

Выделяем доменные зоны, которые отражают бизнес-логику приложения "Зелёный агент".

| Доменная зона | Описание | Ключевые сущности (Глоссарий) |
|---------------|----------|-------------------------------|
| **Пользователи и профили** | Регистрация, авторизация, управление ролями (клиент, курьер, волонтёр), личные кабинеты. | **Клиент** - тот, кто создаёт заказ на вывоз вторсырья. **Курьер** - платный исполнитель. **Волонтёр** - бесплатный исполнитель, работающий за баллы. **Администратор** - проверяет регистрации, решает конфликты. |
| **Заказы и логистика** | Создание, поиск, назначение и отслеживание заказа на вывоз вторсырья. | **Заказ** - запрос на вывоз, содержит тип сырья, объём, адрес, статус. **Статусы:** "новый", "назначен исполнитель", "в пути", "забрано", "доставлено в пункт", "отменён". |
| **Геолокация и карты** | Определение местоположения клиента и курьера, расчёт расстояний, отображение ближайших заказов. | **Точка сбора** - адрес клиента. **Зона поиска** - радиус (по умолчанию 5 км), в котором ищутся исполнители. |
| **Мотивация и баллы** | Начисление, учёт и обмен бонусных баллов для клиентов и волонтёров. | **Бонусный балл** - единица поощрения. **Партнёр** - магазин или сервис, принимающий баллы к оплате. |
| **Приём вторсырья** | Интеграция с пунктами приёма, учёт сданных объёмов, передача данных о доставленном сырье. | **Пункт приёма** - место, куда курьер или волонтёр отвозит вторсырьё. **Фракция** - тип вторсырья (макулатура, стекло, пластик, батарейки). |

## BDD (Behaviour-Driven Development)

**Критический путь MVP:** клиент создаёт заказ на вывоз вторсырья, а курьер принимает и выполняет его.

### Сценарий успеха
> **Дано:**  
> - Клиент зарегистрирован и авторизован в приложении.  
> - У клиента накопилось 5 кг макулатуры и пакет пластиковых бутылок.  
> - Курьер с рейтингом 4.8 находится в 2 км от клиента и свободен.
>
> **Когда:**  
> - Клиент открывает экран создания заказа и выбирает фракции "Макулатура" и "Пластик".  
> - Указывает вес 5 кг и адрес (определён автоматически через геолокацию).  
> - Нажимает "Найти исполнителя".  
> - Система показывает доступных курьеров, клиент выбирает курьера с ценой 200 ₽.  
> - Курьер получает уведомление, принимает заказ и выезжает по адресу.  
> - Курьер забирает вторсырьё, отмечает статус "Забрано".  
> - Курьер отвозит сырьё в пункт приёма и отмечает статус "Доставлено".
> 
> **Тогда:**  
> - Клиент видит на экране статус "Заказ доставлен".  
> - Курьер получает оплату 200 ₽ на свой счёт.  
> - Клиенту начисляются бонусные баллы в соответствии с объёмом сданного сырья.

### Сценарий отказа
> **Дано:**  
> - Клиент зарегистрирован и авторизован.  
> - В радиусе 5 км в данный момент нет ни одного курьера или волонтёра.
> 
> **Когда:**  
> - Клиент заполняет заказ (фракция "Стекло", 3 кг) и нажимает "Найти исполнителя".
>
> **Тогда:**  
> - Система показывает сообщение: "Поблизости пока никого нет. Мы можем уведомить вас, когда появится исполнитель".  
> - Клиенту предлагается подписаться на уведомление или изменить радиус поиска (платная опция не в MVP).  
> - Заказ **не создаётся**, клиент возвращается на главный экран.

## Wireframes

Опишем ключевые экраны, по которым проходит пользователь в критическом пути.

### Экран 1
> **Создание заказа:**
> - Заголовок "Новый заказ"
> - Выпадающий список "Тип вторсырья" (мультивыбор)
> - Поле "Примерный вес, кг"
> - Поле "Адрес" с кнопкой "Определить автоматически"
> - Кнопка "Найти исполнителя" (активная после заполнения)
> - Внизу панель навигации: Главная | Заказы | Профиль

### Экран 2
> **Выбор исполнителя:**
> - Заголовок "Доступные исполнители"
> - Список карточек: имя, рейтинг, расстояние, цена
> - Фильтр: "Курьеры" / "Волонтёры"
> - Кнопка "Выбрать" на каждой карточке
> - Возможность вернуться назад к редактированию заказа

### Экран 3
> **Отслеживание заказа:**
> - Заголовок "Мой заказ"
> - Индикатор статуса: полоса с шагами "Назначен -> В пути -> Забрано -> Доставлено"
> - Текущий статус выделен цветом
> - Информация об исполнителе (имя, телефон)
> - Кнопка "Связаться с исполнителем"

### Экран 4
> **Оценка и завершение:**
> - Заголовок "Заказ выполнен"
> - Детали заказа (фракции, вес, дата)
> - Звёзды для оценки курьера/волонтёра
> - Поле для отзыва (опционально)
> - Кнопка "Завершить"
> - Информация о начисленных баллах

## API-First

Проектируем JSON-контракты для двух главных ручек MVP: создание заказа клиентом и получение списка заказов для курьера.

### Ручка 1: Создание заказа

**Endpoint:**
> `POST v1/orders`

**Input:**
```json
{
  "fraction": "String",
  "weight_kg": "Float",
  "address": "String",
  "lat": "Float",
  "lon": "Float"
}
```

**Output:**
> `200`
```json
{
  "order_id": "String",
  "status": "String",
  "created_at": "DateTime"
}
```
> `400`
```json
{ "error": "Не заполнено поле address" }
```

### Ручка 2: Получение списка доступных заказов (для курьера)

**Endpoint:**
> `GET v1/orders/available?lat=55.75&lon=37.62&radius_km=5`

**Input:**
> Параметры в URL.

**Output:**
> `200`
```json
{
  "orders": [
    {
      "order_id": "String",
      "fraction": "String",
      "weight_kg": "Float",
      "distance_km": "Float",
      "price": "Int",
      "pickup_address": "String"
    }
  ]
}
```
> `400`
```json
{ "error": "Не указаны координаты" }
```
